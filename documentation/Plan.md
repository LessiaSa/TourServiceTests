# План работы по автоматизации тестирования банковского сервиса
## Задача:
Автоматизировать сценарии тестирования сервиса, взаимодействующего с СУБД и API Банка.
## Артефакты:
SUT aqa-shop.jar
## Предусловия:
Тестируемое приложение запускается по инструкции, указанной в в файле Plan.md
# Перед каждым тестом:
1. Открываем страницу http://localhost:8080 в браузере
2. Открываем приложение Docker Desctop на компьютере
3. Открываем приложение для работы с СУБД на компьютере (например DBeaver)
4. Запускаем Docker  в приложении IntelliJIDEA с помощью команды docker compose up
5. Запускаем Jar -файл с помощью команды  java -jar ./artifacts/aqa-shop.jar -P:jdbc.url=jdbc:mysql://localhost:8080/db

# Тестовые сценарии:
## Позитивные сценарии тестирования в кредитном сервисе Credit Gate:
### 1. Успешная обработка данных в кредитном сервисе CreditGate при вводе валидных данных русскими буквами в поле "Владелец"
#### Шаги:
  1. Нажать кнопку "Купить в кредит"
  2. В поле ввода "Номер карты" ввести валидные данные - 1111 2222 3333 4444
  3. В поле "Месяц" ввести валидные данные - следующий месяц
  4. В поле "Год" ввести валидные данные - следующий год
  5. В поле "Владелец" ввести валидные данные - на кириллице - имя и фамилию
  6. В поле CVC/CVV ввести валидное значение - три цифры
  7. Нажать кнопку "Продолжить"
#### Ожидаемый результат: 
появляется всплывающее окно с сообщением:"Успешно! Операция одобрена Банком."<br>
В БД есть запись APPROVED

4. JUnit 5- одна из самых популярных платформ модульного тестирования

7. MySQL 8.0.18 (заявлена поддержка этой СУБД)
8. Java Faker-библиотека, которую можно использовать для создания широкого спектра реально выглядящих данных от адресов до ссылок на популярные культуры.
9. Google Chrome 88.0.4324.104-один из самых популярных и удобных браузеров.
10. GitHub-крупнейший веб-сервис для хостинга IT-проектов и их совместной разработки, основанный на системе контроля Git.


# Перечень и описание возможных рисков при автоматизации
1. Отсутствие документации и ТЗ, без которых непонятно - как именно должна работать SUT и какое именно поведение системы считается ошибочным
2. Реальная система будет отличаться от SUT
3. Использование захардкоженых номеров карт может привести к ошибкам в работе реальной системы
4. Отсутствие test_id  в css-селекторах усложняет тестирование

# Интервальная оценка с учетом рисков в часах
1. Создание проекта и запуск SUT, Docrer и DBeaver - 15 часов (не всегда все гладко запускается с первого раза, возможны ошибки и недочеты)
2. Написание тест-плана - 8 часов,
3. Написание, прогон и проверка автотестов - 24 часа (Возможны ошибки в коде и при запуске автотестов - потребуется время на их исправление),
4. Дополнительное время на поиск и исправление ошибок - 24 часа
5. Подготовка отчетов по итогам тестирования - 12 часов
6. Подготовка отчетных документов по итогам тестирования - 10 часов
7. Подготовка отчетных документов по итогам автоматизации -10 часов


# План сдачи работ
1. Написание и прогон автотестов - 3 дня с учетом возможных рисков после утверждения плана руководителем
2. Написание отчетных документов - 2 дня после утверждения руководителем автотестов
